Parking Lot System Design

Smart Parking Management System
System Design Interview Document

1. Requirements Gathering

1.1 Problem Statement

Design a smart parking lot management system that can handle multiple parking lots, different vehicle types, real-time spot tracking, and automated billing.

1.2 Clarifying Questions

  • Scale: How many parking lots? Spots per lot? Daily vehicles?
  • Vehicle Types: Cars, motorcycles, trucks, handicapped spots?
  • Payment: Hourly, daily, monthly passes, different rates?
  • Entry/Exit: Automated gates, ticket systems, mobile app?
  • Reservations: Can users reserve spots in advance?
  • Real-time Updates: Live availability updates needed?
  • Analytics: Usage patterns, revenue reporting?
  • Integration: Third-party navigation apps, payment gateways?

1.3 Functional Requirements

  1. Spot Management: Track availability of parking spots
  2. Vehicle Entry/Exit: Automated entry and exit processing
  3. Payment Processing: Multiple payment methods and pricing
  4. Reservations: Advance booking of parking spots
  5. Real-time Updates: Live availability information
  6. User Management: Customer accounts and preferences
  7. Notifications: SMS/email notifications for bookings
  8. Reporting: Usage analytics and revenue reports
  9. Admin Dashboard: Management interface for operators

3. Database Design

3.1 Core Tables Schema

Parking Lots

CREATE TABLE parking_lots ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), name VARCHAR(255) NOT NULL, address TEXT NOT NULL, latitude DECIMAL(10, 8), longitude DECIMAL(11, 8), total_spots INTEGER NOT NULL, operating_hours JSONB, -- {"monday": "00:00-23:59", ...} pricing_config JSONB, amenities JSONB, -- ["ev_charging", "covered", "security"] status lot_status DEFAULT 'active', created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, INDEX idx_location (latitude, longitude), INDEX idx_status (status) ); CREATE TYPE lot_status AS ENUM ('active', 'maintenance', 'closed');

Parking Spots

CREATE TABLE parking_spots ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_id UUID NOT NULL REFERENCES parking_lots(id), spot_number VARCHAR(50) NOT NULL, floor_level INTEGER DEFAULT 1, vehicle_type vehicle_type NOT NULL, status spot_status DEFAULT 'available', is_handicapped BOOLEAN DEFAULT false, is_ev_charging BOOLEAN DEFAULT false, location_metadata JSONB, -- {"section": "A", "row": 1, "column": 15} created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, UNIQUE(lot_id, spot_number), INDEX idx_lot_status (lot_id, status), INDEX idx_lot_type_status (lot_id, vehicle_type, status) ); CREATE TYPE vehicle_type AS ENUM ('car', 'motorcycle', 'truck', 'handicapped'); CREATE TYPE spot_status AS ENUM ('available', 'occupied', 'reserved', 'maintenance');

Parking Sessions

CREATE TABLE parking_sessions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_id UUID NOT NULL REFERENCES parking_lots(id), spot_id UUID NOT NULL REFERENCES parking_spots(id), user_id UUID REFERENCES users(id), license_plate VARCHAR(20) NOT NULL, vehicle_type vehicle_type NOT NULL, -- Entry details entry_time TIMESTAMP NOT NULL, entry_gate_id VARCHAR(50), ticket_id VARCHAR(100) UNIQUE, -- Exit details (nullable until exit) exit_time TIMESTAMP, exit_gate_id VARCHAR(50), -- Billing reservation_id UUID REFERENCES reservations(id), hourly_rate DECIMAL(6,2), total_cost DECIMAL(8,2), payment_status payment_status DEFAULT 'pending', payment_id VARCHAR(255), created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, INDEX idx_lot_entry_time (lot_id, entry_time), INDEX idx_license_plate (license_plate), INDEX idx_ticket_id (ticket_id), INDEX idx_active_sessions (lot_id, exit_time) WHERE exit_time IS NULL ); CREATE TYPE payment_status AS ENUM ('pending', 'paid', 'failed', 'refunded');

Reservations

CREATE TABLE reservations ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), lot_id UUID NOT NULL REFERENCES parking_lots(id), spot_id UUID REFERENCES parking_spots(id), user_id UUID NOT NULL REFERENCES users(id), license_plate VARCHAR(20) NOT NULL, vehicle_type vehicle_type NOT NULL, start_time TIMESTAMP NOT NULL, end_time TIMESTAMP NOT NULL, duration_hours INTEGER GENERATED ALWAYS AS (EXTRACT(epoch FROM (end_time - start_time))/3600) STORED, status reservation_status DEFAULT 'confirmed', total_cost DECIMAL(8,2) NOT NULL, payment_id VARCHAR(255), -- QR code and entry details entry_code VARCHAR(20) UNIQUE, qr_code_url VARCHAR(500), created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, cancelled_at TIMESTAMP, INDEX idx_user_start_time (user_id, start_time), INDEX idx_lot_time_range (lot_id, start_time, end_time), INDEX idx_active_reservations (lot_id, status, start_time, end_time) WHERE status IN ('confirmed', 'checked_in') ); CREATE TYPE reservation_status AS ENUM ('confirmed', 'checked_in', 'completed', 'cancelled', 'no_show');

3.2 Indexing Strategy

-- Composite indexes for common queries CREATE INDEX idx_spots_availability ON parking_spots (lot_id, status, vehicle_type) WHERE status = 'available'; -- Partial index for active sessions CREATE INDEX idx_active_parking_sessions ON parking_sessions (lot_id, spot_id, entry_time) WHERE exit_time IS NULL; -- Time-based partitioning for sessions (monthly) CREATE INDEX idx_sessions_time_partition ON parking_sessions (DATE_TRUNC('month', entry_time), lot_id); -- Geospatial index for nearby lots CREATE INDEX idx_lots_location ON parking_lots USING gist(ll_to_earth(latitude, longitude));

3.3 Data Partitioning

Partitioning Strategy:
  • Horizontal: Partition parking_sessions by month for better query performance
  • Sharding: Distribute data by geographic regions (city-based shards)
  • Read Replicas: Analytics and reporting queries use read-only replicas
  • Archival: Sessions older than 2 years moved to cold storage

5. Deep Dive - Real-Time Spot Tracking

5.1 Spot State Management

Critical Challenge: Maintaining accurate real-time spot availability across multiple data sources (sensors, cameras, manual updates) while handling hardware failures and network issues.

State Synchronization Architecture

@Service public class SpotStateManager { @Autowired private RedisTemplate<String, String> redis; @Autowired private SpotRepository spotRepository; @Autowired private KafkaTemplate<String, Object> kafka; public void updateSpotStatus(String spotId, SpotStatus newStatus, String source, long timestamp) { String lockKey = "spot_lock:" + spotId; RLock lock = redisson.getLock(lockKey); try { if (lock.tryLock(2, 10, TimeUnit.SECONDS)) { // Get current state from cache SpotState currentState = getSpotState(spotId); // Validate transition if (!isValidTransition(currentState.getStatus(), newStatus)) { log.warn("Invalid state transition for spot {}: {} -> {}", spotId, currentState.getStatus(), newStatus); return; } // Check for conflicting updates (timestamp comparison) if (timestamp < currentState.getLastUpdated()) { log.info("Ignoring outdated update for spot {}", spotId); return; } // Update cache immediately SpotState newState = SpotState.builder() .spotId(spotId) .status(newStatus) .source(source) .lastUpdated(timestamp) .build(); updateSpotCache(newState); // Async database update CompletableFuture.runAsync(() -> { updateSpotDatabase(spotId, newStatus, timestamp); }); // Publish event for real-time notifications publishSpotStatusEvent(spotId, currentState.getStatus(), newStatus, source); } } catch (InterruptedException e) { Thread.currentThread().interrupt(); throw new SpotUpdateException("Failed to acquire lock for spot: " + spotId); } finally { if (lock.isHeldByCurrentThread()) { lock.unlock(); } } } }

Sensor Data Integration

@Component public class SensorDataProcessor { @KafkaListener(topics = "sensor.occupancy.updates") public void processSensorUpdate(SensorMessage message) { try { // Validate sensor data if (!isValidSensorReading(message)) { log.warn("Invalid sensor reading: {}", message); return; } // Determine spot status from sensor data SpotStatus status = message.getDistance() < OCCUPANCY_THRESHOLD ? SpotStatus.OCCUPIED : SpotStatus.AVAILABLE; // Update spot state with sensor source spotStateManager.updateSpotStatus( message.getSpotId(), status, "sensor:" + message.getSensorId(), message.getTimestamp() ); } catch (Exception e) { log.error("Error processing sensor update: {}", message, e); // Send to dead letter queue for manual review kafkaTemplate.send("sensor.updates.dlq", message); } } @Scheduled(fixedRate = 30000) // Every 30 seconds public void checkSensorHealth() { List<String> inactiveSensors = findInactiveSensors(); for (String sensorId : inactiveSensors) { // Mark associated spots as "sensor_offline" List<String> affectedSpots = getSpotsBySensor(sensorId); for (String spotId : affectedSpots) { spotStateManager.markSpotSensorOffline(spotId, sensorId); } // Send alert to operations team alertService.sendSensorOfflineAlert(sensorId); } } }

5.2 License Plate Recognition Integration

@Service public class LicensePlateService { @Autowired private OpenALPRClient alprClient; @EventListener public void handleVehicleEntry(VehicleEntryEvent event) { // Process camera image for license plate CompletableFuture<String> plateRecognition = CompletableFuture.supplyAsync(() -> { try { return alprClient.recognizePlate(event.getImageData()); } catch (Exception e) { log.error("Plate recognition failed", e); return null; } }); // Timeout after 5 seconds try { String licensePlate = plateRecognition.get(5, TimeUnit.SECONDS); if (licensePlate != null) { // Check for existing reservation Optional<Reservation> reservation = findActiveReservation(licensePlate); if (reservation.isPresent()) { // Automatic entry for reserved spot processReservedEntry(reservation.get(), event.getGateId()); } else { // Walk-in customer processWalkInEntry(licensePlate, event.getGateId()); } } else { // Fallback to manual ticket system issueManualTicket(event.getGateId()); } } catch (TimeoutException e) { log.warn("Plate recognition timeout, issuing manual ticket"); issueManualTicket(event.getGateId()); } } }

5.3 Real-Time Updates via WebSocket

@Controller public class ParkingWebSocketController { @Autowired private SimpMessagingTemplate messagingTemplate; @EventListener public void handleSpotStatusChange(SpotStatusChangeEvent event) { // Send to specific lot subscribers String destination = "/topic/lots/" + event.getLotId() + "/updates"; SpotUpdateMessage message = SpotUpdateMessage.builder() .spotId(event.getSpotId()) .spotNumber(event.getSpotNumber()) .oldStatus(event.getOldStatus()) .newStatus(event.getNewStatus()) .timestamp(event.getTimestamp()) .build(); messagingTemplate.convertAndSend(destination, message); // Send lot capacity update LotCapacityUpdate capacityUpdate = calculateLotCapacity(event.getLotId()); messagingTemplate.convertAndSend( "/topic/lots/" + event.getLotId() + "/capacity", capacityUpdate ); } @MessageMapping("/lots/{lotId}/subscribe") @SendTo("/topic/lots/{lotId}/updates") public String subscribToLotUpdates(@DestinationVariable String lotId, Principal principal) { // Authenticate and authorize subscription if (!hasAccessToLot(principal.getName(), lotId)) { throw new AccessDeniedException("No access to lot: " + lotId); } // Send current lot state return getCurrentLotState(lotId); } }

5.4 Caching Strategy

Multi-Level Caching Approach:
  • L1 Cache: Application memory for frequently accessed spots
  • L2 Cache: Redis for real-time spot states and availability
  • L3 Cache: Database query result caching
  • TTL Strategy: Spot states (5min), Availability counts (1min), Static data (1hr)
@Service public class ParkingCacheService { @Cacheable(value = "lot_availability", key = "#lotId", unless = "#result == null") public LotAvailability getLotAvailability(String lotId) { return calculateRealTimeAvailability(lotId); } @CacheEvict(value = "lot_availability", key = "#lotId") public void invalidateLotAvailability(String lotId) { // Auto-called when spot status changes } @Scheduled(fixedRate = 60000) // Every minute public void refreshAvailabilityCounts() { List<String> activeLots = getActiveLotIds(); for (String lotId : activeLots) { // Pre-calculate and cache availability LotAvailability availability = calculateRealTimeAvailability(lotId); redisTemplate.opsForValue().set( "availability:" + lotId, availability, Duration.ofMinutes(2) ); } } }

8. Disaster Recovery & High Availability

8.1 Multi-Region Architecture

Regional Deployment Strategy:
  • Primary Regions: US-West, US-East, EU-West
  • Failover Time: < 30 seconds for automated failover
  • Data Sync: Near real-time replication for critical data
  • Hardware Integration: Local processing with cloud backup

8.2 Backup & Recovery

  • Database Backups: Continuous WAL shipping, daily snapshots
  • Application State: Stateless services with external state
  • Hardware Redundancy: Backup sensors and gate controllers
  • Network Resilience: Multiple ISP connections, cellular backup

8.3 Business Continuity

Critical Failure Scenarios:
  • Total System Outage: Manual ticket operation within 5 minutes
  • Payment System Failure: Deferred billing with exit logging
  • Network Isolation: Local caching supports 2-hour operation
  • Gate Hardware Failure: Manual override and mobile checkout
